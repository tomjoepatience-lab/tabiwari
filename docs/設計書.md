# 旅行・イベント割り勘アプリ 設計書

> 仮称: **TabiWari（タビワリ）** — 旅行/イベント単位のレシート割り勘＆支出分析アプリ
> 第10回「テーマと設計」課題の成果物。機能要件 → 設計（フロント体験 / バック処理 / 永続化）の順でまとめる。

---

## 1. コンセプト

旅行・イベントごとに支出を記録し、**レシートの明細単位で「誰の負担か」を割り当てて精算**する Web アプリ。

- 着想元: 自作モバイルアプリ `tabikake` の「支出をイベント（旅行/デート）単位でまとめて振り返る」構造。
- tabikake との違い: 記録の単位が「1支出＝1ピン」ではなく **「1レシート＝複数明細、明細ごとに負担者」**。割り勘がメイン。
- 解きたい課題: 旅行の精算が面倒（「これ誰の？」「結局いくら返す？」）／旅行ごとにいくら使ったか後で分からない。
- 提供する体験: レシートを取り込み→明細に名前をポチポチ割り当て→**人ごとの負担額と精算**が自動で出る。分析タブで旅行横断の支出も見える。
- **実体の単位 = 1つの思い出**: 「店名・カテゴリ・写真・場所（ピン）・明細＋割り勘」をまとめた1レシートが、思い出タブの1カード／地図の1ピンになる。
- **オリジナリティ（tabikake 由来）**:
  - **写真（デフォルト）**: 記録に写真を付け、**思い出タブ**に写真カードで残る。カード or 地図ピンをクリック→**大きい写真＋割り勘詳細**。
  - **地図ピン（Leaflet＋OSM）**: ピンは「訪れた場所」を表し、店名・カテゴリ・写真と連動。「どこで何にいくら使ったか」を地図で振り返れる。

## 2. 機能要件（マイルストーン①）

### スコープと開発段階

| 段階 | ゴール | 機能 |
|---|---|---|
| **M0 土台** | 3層が繋がって動く | Express＋静的配信、DB接続、旅行の作成、メンバー登録 |
| **M1 手入力MVP** | 割り勘が成立する | レシートを手入力（店名・カテゴリ・日付・支払者・明細）、各明細に負担者を複数選択、**写真を添付**、旅行ごとの合計 |
| **M2 思い出＆地図＆分析** | 振り返れる | **思い出タブ（写真カード一覧）→クリックで大きい写真＋割り勘詳細**、**地図ピン（店名・カテゴリ・写真と連動）**、人ごとの負担額、精算（誰が誰にいくら）、グラフ |
| **M3 レシートOCR** | 入力を楽に | レシート画像→Tesseract.js で明細を下書き抽出→ユーザーが修正＆負担者割当 |
| **M4 拡張枠** | 差別化・任意 | **思い出アルバム自動生成**、カテゴリ自動分類、ユーザー認証/共有、CSVエクスポート、AI相談 |

> **M0→M1 が提出の最低ライン。M2 まで行けば「使える割り勘アプリ」。OCR(M3)は手入力が動いてから上に乗せる**（先に作ると沼る）。

### 割り切り（非機能）
- 通貨は日本円のみ・金額は正の整数（小数なし）。
- M1〜M3 はシングルユーザー前提（認証なし）。`users` は M4 で追加できる設計にする。
- OCR は「完全自動」を狙わない。**下書き生成＋手修正**が前提。

## 3. 設計（マイルストーン②）

### 3-1. インプット / アウトプット（フロントでの体験）

HTML / CSS / TypeScript（素の DOM）。画面は3つ。

| 画面 | インプット（操作） | アウトプット（表示） |
|---|---|---|
| **旅行一覧 / 分析** | 旅行を選ぶ・新規作成 | 旅行カード（期間・総額）、旅行横断の支出グラフ |
| **旅行詳細** | レシート追加、メンバー編集、**思い出/地図/リストの切替** | **思い出タブ＝写真カード一覧**、地図タブ＝ピン、リストタブ＝レシート、旅行の総額・**人ごとの負担額**・精算結果 |
| **詳細モーダル** | 写真カード or 地図ピンをクリック | **大きい写真＋割り勘詳細**（店名・カテゴリ・明細ごとの負担者・合計・誰の負担いくら） |
| **レシート編集** | 店名・カテゴリ・支払者・明細入力、各明細に負担者を複数選択、**写真を選択**、現在地取得 or 地図クリックで位置指定 | 合計・1人あたりの按分プレビュー、写真プレビュー、地図上の仮ピン |

**体験のポイント**
- 明細の負担者はチェックボックス（メンバー分のボタン）で選ぶ → 選んだ人数で等分。
- 入力後は変更箇所の DOM だけ更新（一覧が長くてもスクロールが飛ばないように）。
- 精算は「誰が誰にいくら払えば終わるか」を1行ずつ提示。
- 思い出タブ・地図ピンは同じ実体（レシート）を別の見せ方で表示し、どちらをクリックしても同じ詳細モーダルへ。
- グラフは Chart.js、地図は Leaflet.js（OSM タイル）を使用。
- **位置の取得（記録時）**: ①ブラウザの Geolocation API で現在地、②地図をクリックして指定、のどちらか。緯度経度は任意（無い場合はピンを置かない）。
- **写真（記録時）**: `<input type=file accept=image/*>` で選択 → **アップロード前にブラウザで縮小**（canvas で長辺〜1280px・JPEG）して送る（DB/転送を軽く保つ）。

### 3-2. プロセス（バックエンドの処理）

Node.js + Express の REST API。「リクエスト受信 → SQL 実行 → JSON 返却」。

| メソッド | パス | 処理 |
|---|---|---|
| GET | `/api/trips` | 旅行一覧（各旅行の総額付き） |
| POST | `/api/trips` | 旅行を作成 |
| GET | `/api/trips/:id` | 旅行詳細（メンバー・レシート・集計・精算） |
| POST | `/api/trips/:id/members` | メンバー追加 |
| POST | `/api/trips/:id/receipts` | レシート登録（明細＋負担者まとめて） |
| PUT | `/api/receipts/:id` | レシート更新 |
| DELETE | `/api/receipts/:id` | レシート削除 |
| GET | `/api/receipts/:id/photo` | レシート写真本体を配信（案A: BYTEA を返す） |
| GET | `/api/trips/:id/settlement` | 精算結果（誰が誰にいくら） |
| GET | `/api/trips/:id`（内包） | 思い出写真一覧（`photos`）も同梱して返す |
| POST | `/api/trips/:id/photos` | 思い出写真を1枚追加（写真＋キャプション＋撮影日） |
| PUT | `/api/trip-photos/:id` | 思い出写真のメタ更新（キャプション・撮影日・並び順） |
| DELETE | `/api/trip-photos/:id` | 思い出写真を削除 |
| GET | `/api/trip-photos/:id/photo` | 思い出写真本体を配信（BYTEA） |
| POST | `/api/auth/register` / `login` / `logout` | 登録・ログイン・ログアウト（cookieセッション） |
| GET | `/api/auth/me` | 現在のユーザー＋所属グループ |
| GET/POST | `/api/groups` / `POST /api/groups/join` | グループ一覧・作成・招待コードで参加 |

**負担額・精算ロジック（サーバ側で計算）**
1. 各明細の金額を、割り当てメンバー数で等分 → メンバーごとに「負担額」を合算。
2. レシートの `paid_by`（実際に払った人）ごとに「支払額」を合算。
3. 各メンバーの収支 = 支払額 − 負担額（プラス＝もらう側、マイナス＝払う側）。
4. プラス側とマイナス側を貪欲法でマッチング → 「A→B にXX円」の最小本数を提示。

**OCR（M3）の処理イメージ**
- 画像アップロードはフロントで Tesseract.js を実行 → 生テキストを取得。
- 正規表現で「商品名 …… 金額」っぽい行を抽出して**明細の下書き**を作る。
- 下書きをフォームに流し込み、ユーザーが修正・負担者割当して保存（API は手入力と同じ）。

**位置→店名（記録時・任意）**
- Geolocation で取得した緯度経度を OSM Nominatim の逆ジオコーディング（無料・キー不要）に投げ、`place_name` の候補にする。

**写真の保存（要注意）**
- Render は**ファイルシステムが揮発性**（再デプロイで画像が消える）。永続化に2案：
  - **A（推奨・MVP）**: 縮小済み画像を PostgreSQL に `BYTEA` で保存。外部サービス不要で課題向き。`GET /api/receipts/:id/photo` で配信。
  - **B（スケール時）**: Render Persistent Disk か外部ストレージ（S3/Cloudinary 等）に保存し、`image_url` に URL を持つ。
- まず A で実装。`receipts.image_url` は B 用に残し、A の間は内部配信エンドポイントを指す。

**バリデーション（サーバでも必ず）**
- amount/price > 0、type/日付の妥当性、`paid_by`・負担者が同じ旅行のメンバーか。
- SQL はパラメータ化クエリ（`$1, $2`）でインジェクション対策。

### 3-3. 永続化するデータ（PostgreSQL）

```sql
CREATE TABLE trips (
  id          SERIAL PRIMARY KEY,
  title       TEXT NOT NULL,
  kind        TEXT NOT NULL DEFAULT 'trip', -- 'trip'=旅行 / 'daily'=日常（普段使い）
  start_date  DATE,
  end_date    DATE,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE members (
  id       SERIAL PRIMARY KEY,
  trip_id  INTEGER NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
  name     TEXT NOT NULL
);

CREATE TABLE receipts (
  id           SERIAL PRIMARY KEY,
  trip_id      INTEGER NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
  store_name   TEXT,
  purchased_on DATE NOT NULL,
  paid_by      INTEGER REFERENCES members(id),   -- 実際に払った人
  category     TEXT,                              -- 食費/交通/宿泊など（任意）
  lat          DOUBLE PRECISION,                  -- 地図ピン用・緯度（任意）
  lng          DOUBLE PRECISION,                  -- 地図ピン用・経度（任意）
  place_name   TEXT,                              -- 逆ジオコーディング結果（任意）
  photo        BYTEA,                             -- 写真本体（案A・MVP。縮小済みJPEG）
  image_url    TEXT,                              -- 写真URL（案B用。MVPは内部配信エンドポイントでも可）
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE items (
  id          SERIAL PRIMARY KEY,
  receipt_id  INTEGER NOT NULL REFERENCES receipts(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,
  price       INTEGER NOT NULL CHECK (price > 0),
  quantity    INTEGER NOT NULL DEFAULT 1
);

-- 「この明細を誰が負担するか」。複数行＝割り勘（選んだ人数で等分）
CREATE TABLE item_shares (
  item_id   INTEGER NOT NULL REFERENCES items(id) ON DELETE CASCADE,
  member_id INTEGER NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  PRIMARY KEY (item_id, member_id)
);

-- 思い出写真（レシートとは別。アルバムの素材はこれだけ）
CREATE TABLE trip_photos (
  id          SERIAL PRIMARY KEY,
  trip_id     INTEGER NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
  photo       BYTEA NOT NULL,             -- 縮小済みJPEG
  caption     TEXT,                       -- ひとことコメント（任意）
  taken_on    DATE,                       -- 撮影日（任意・アルバムの並び順）
  sort_order  INTEGER NOT NULL DEFAULT 0, -- 手動並べ替え用（編集フェーズ）
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**プロジェクト化＋ダッシュボード（方針追加）**
- `trips` を「**プロジェクト（家計簿）**」として一般化し、`kind`（`trip`=旅行 / `daily`=日常）を持たせた。
- トップ画面は**ダッシュボード**: プロジェクトを 旅行 / 日常 に分けてカード表示 → 押すとそのプロジェクトに入る。`GET /api/analytics` を**統括（全プロジェクト横断）**として併置。
- 詳細は `kind` で出し分け: 旅行＝思い出/地図/リスト＋アルバム＋期間、日常＝リスト＋月次・カテゴリ分析（割り勘・集計・精算は両方で利用可）。

**共有家計簿（認証＋グループ）**
- ユーザー名＋パスワード（scrypt）＋**cookieセッション**（`sessions` テーブルの不透明トークン、HttpOnly）。外部ライブラリ不使用（Node `crypto` と素のcookie操作）。
- `user_groups`（家族の共有単位・招待コード付き）＋ `group_members`。**プロジェクト（trips）は `group_id` で1グループに属する**。
- 全プロジェクト系APIは `requireAuth`＋**所属グループでスコープ**（自分の所属グループのデータしか読めない／作れない）。家族は招待コードで同じグループに参加 → 共有家計簿。
- デモ: `demo` / `demo1234`（seed 投入時）。

**レシートとアルバムの分離（方針変更）**
- 当初は「1レシート＝1思い出」で、レシート写真をそのままアルバム素材にする想定だった。
- 変更後: **レシート写真は「レシート管理」（リストタブのサムネ＋詳細モーダル）に残すだけ**で、割り勘の証拠として扱う。
- **アルバムの素材は `trip_photos`（思い出写真）のみ**。思い出タブから旅行ごとに写真をアップロードし、それを時系列に組版して表紙＋ページめくりで見せる（tabikake のアルバム機能を移植）。

**設計の肝**
- `item_shares` を多対多にすることで「自分だけの弁当＝1人」「みんなで割るおつまみ＝全員」を**同じ仕組み**で表現。
- `paid_by`（払った人）と `item_shares`（負担する人）を分けて持つから、精算（誰が誰に返すか）が計算できる。

## 8. 次のステップ

1. `package.json` ＋ Express＋静的配信の最小サーバ（M0）
2. `schema.sql` で trips/members/receipts/items/item_shares を作成
3. 旅行作成・メンバー登録 → レシート手入力（明細＋負担者）＋**写真添付**→ 旅行ごとの合計（M1）
