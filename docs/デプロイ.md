# デプロイ移行: Render → Vercel

kakeibo（マネコ家計簿）を常駐サーバー(Render)からサーバーレス(Vercel)へ移行するための手順書。
**並行稼働期間があるため、この移行作業中も Render 側は動かしたまま**にする（Render を止めるのは
Vercel 側の検証が終わってから）。

## 前提
- ユーザーは **Vercel Pro に加入済み**（商用利用可。Hobby プランの商用利用不可の制約は対象外）。
- DB は Neon（PostgreSQL）を使い続ける。`DATABASE_URL` の中身を差し替えるだけで Render/Vercel どちらからも同じ Neon に繋がる。

## コード側の変更（実装済み）
- `src/server/app.ts`: Express アプリの組み立てのみを担当（`app.set('trust proxy', 1)` 済み）。
- `src/server/index.ts`: `app.ts` を import して `listen` するだけ（Render・ローカル`npm run dev`用）。
- `api/index.ts`: Vercel サーバーレス関数のエントリ。`src/server/app.ts` の Express アプリをそのまま default export。
- `vercel.json`（リポジトリルート）: `buildCommand`（クライアントのみビルド）・`rewrites`（`/api/*` → `/api/index`）・`functions`（`api/index.ts` に `maxDuration: 60`）。
- `src/server/db.ts`: `PG_POOL_MAX` 環境変数でプールの最大接続数を絞れるように（既定10、未設定なら従来通り）。

## ユーザーがやること（vercel.com 側）
1. https://vercel.com → **Add New... → Project**
2. GitHub の `tabiwari` リポジトリを Import
3. **Framework Preset: 「Other」** を選択（自動検出に任せない）
4. **Environment Variables** に以下を設定
   - `DATABASE_URL` … **Neon の Pooled connection**（Neon ダッシュボード → 該当プロジェクト →
     **Connection string** → **「Pooled connection」** を選んでコピー。`-pooler` が付いたホスト名になっているものを使うこと。
     Direct connection ではなく Pooled を使う理由: サーバーレス関数は同時に多数のインスタンスが
     立ち上がり得るため、PgBouncer 経由の pooled 接続でないと Neon 側の同時接続数上限にすぐ達する）
   - `GEMINI_API_KEY` … OCR用（既存のRenderと同じ値でOK）
   - `ANTHROPIC_API_KEY` … OCRのフォールバック用（既存のRenderと同じ値でOK）
   - （任意）`PG_POOL_MAX` … 未設定でOK（既定10）。接続数が気になる場合は `3`〜`5` 程度に絞ってよい
   - （任意）`GOOGLE_MAPS_API_KEY` … 地図を使う場合
5. **Deploy** をクリック

Build Command / Output Directory は `vercel.json` の `buildCommand`（`npm run build:client`）に従うため
プロジェクト設定側で上書きしなくてよい。**静的ファイル（`public/` 配下の index.html・style.css・icons等）は
Vercel が `public/` を自動で静的配信するプロジェクト設定を前提にしている**（`vercel.json` 側には静的配信の
設定を書いていない）。もし配信されない場合は Vercel の Project Settings → Build & Development Settings で
Output Directory が `public` になっているか確認すること。

## 検証観点（fableが後で実施）
デプロイ後、発行された `*.vercel.app` のURLに対して以下を確認する:
- ログイン（新規登録・ログイン・ログアウト・セッション維持）
- 記録（支出の追加・編集・削除、レポート集計）
- OCR（レシート画像アップロード→明細抽出。1日5回制限も含む）
- 画像（レシート画像・思い出写真のアップロード/表示）
- 地図（地図ピンの表示、Google Maps キーが効いているか）

## 切替後の作業（このタスクではやらない）
- `expo/App.tsx` の `source.uri` を Vercel の URL（独自ドメインがあればそちら）に差し替え
- `capacitor.config.ts`（M1予備の Capacitor シェル）の URL も同様に差し替え
- Render の停止は **Vercel側の検証が完了してから**（並行稼働期間を経て切替）

## 注意点
- **リクエストボディ上限**: Vercel のサーバーレス関数はリクエストボディに **4.5MB** のプラットフォーム上限がある
  （Hobby/Pro 共通）。`express.json({ limit: '8mb' })` はRender向けの設定のままだが、実運用の画像は
  クライアント側でリサイズ済みのため通常はこの上限内に収まる想定。極端に大きい画像の直アップロードでは
  413エラーになり得る点は留意。
- **`maxDuration`**: `vercel.json` では OCR の60秒タイムアウトに合わせて `60`(秒) を設定している。
  Vercel Pro プランでは最大 `300`(秒) まで引き上げ可能なので、OCRやその他の処理でタイムアウトが
  頻発するようなら `functions.maxDuration` を増やす選択肢がある。
- **インメモリのレート制限**: `src/server/api.ts` の `joinHits` / `claimHits`（親子連携コード入力・お手伝い申請の
  1分10回制限）はプロセス内メモリで管理しており、サーバーレスでは関数インスタンスが複数立ち上がると
  インスタンス毎にカウントがリセットされ実質的に緩くなる。実害の小さい簡易ガードのため許容し、DB化などの
  対応はしない。
- **ログの重複**: `src/server/ocr.ts` の「初回だけ出す」プロバイダ／フォールバックログは、サーバーレスでは
  関数インスタンスの数だけ出ることがある（実害なし、対処不要）。
- Neon は無通信でサスペンドすることがあるが、`src/server/db.ts` 側で pool のエラーを握りつぶして
  次クエリで自動再接続する既存の仕組みがあるため、Vercel環境でも同様に動作する想定。
